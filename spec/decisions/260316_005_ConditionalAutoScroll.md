# 260316_005 — 條件式自動捲動 + 返回底部按鈕

> 日期：2026-03-16  
> 狀態：已採納  
> 取代：`main_ide.md` v1.11.0 的「自動捲動行為」（無條件滾動規格）

---

## 背景

原規格（v1.11.0）的自動捲動為無條件觸發：每次新訊息或串流 chunk 都強制將視窗拉到最底部。  
實際使用中，使用者在 AI 回應過程中往上捲動閱讀歷史訊息時，視角會被強制拉回，造成閱讀中斷。  
Request #21（發起：osw-myai-app）要求改為條件式捲動，並新增懸浮「返回底部」按鈕。

---

## 決策一：條件式自動捲動，以「接近底部」狀態作為開關

**決策**：新增 `isAtBottom` 狀態，僅在使用者位於底部（距底部 ≤ 56dp）時才執行自動捲動。

**決策過程**：
- 閾值候選：Issue 建議 48–64dp，取中間值 **56dp**（等同於 Material 3 FAB 高度），既不會因 1px 偏差誤判，也不會在訊息較短時誤觸。
- `isAtBottom` 在 ScrollController 的 scroll listener 中即時更新，不依賴 frame 後回呼，確保狀態反應即時。

**其他考慮**：
- 若以「是否正在主動捲動」作為判斷（`ScrollNotification`），實作較複雜，且在 Flutter 慣性捲動後狀態難以穩定。
- 以位置閾值判斷最簡單可靠。

---

## 決策二：送出訊息 / 切換角色時強制重置 `isAtBottom = true`

**決策**：以下兩個場景強制捲到底部，並同時重置 `isAtBottom = true`：
1. 使用者送出新訊息（使用者的主動意圖就是看最新回應）
2. 切換至不同角色（角色切換後從各角色最後狀態開始）

**決策過程**：
- 兩個場景的共同點：使用者有明確意圖「切換到最新狀態」，此時強制捲動是符合預期的行為，不構成干擾。
- 若切換角色後不重置，上一個角色的捲動位置可能誤留在新角色，導致新角色看不到最新訊息。

---

## 決策三：返回底部按鈕用 `AppAssistChip`，本期不加未讀 Badge

**決策**：按鈕使用 `AppAssistChip`（含 `↓` leading icon，文字「返回底部」），懸浮於訊息列表右下角，本期**不加未讀計數 Badge**。

**決策過程**：
- Request #21 評估未讀 Badge 為 nice-to-have，本次不強制要求。
- `AppAssistChip` 已是 Design System 定義元件，使用成本低，外觀符合 Material 3 風格。
- 若未來確認需要未讀計數（例如 AI 回應中使用者往上看，回來後顯示新增幾則），由 app 角色發 Request 補規格。

---

## 影響

- `main_ide.md` → 「自動捲動行為」改為「條件式自動捲動」（v1.11.4）
- App 實作：需新增 `isAtBottom` 狀態管理 + ScrollController listener + 懸浮按鈕元件
- 無 API / 資料層影響
