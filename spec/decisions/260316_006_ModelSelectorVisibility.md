# 260316_006 — 模型選擇器在模型清單取得前不顯示

> 日期：2026-03-16
> 發起：osw-myai-app（Request #22）
> 狀態：已採用

---

## 問題背景

原規格規定模型選擇器「常駐顯示」，且在尚未連線或 `GET /models` 取得失敗時使用靜態 fallback 清單（`claude-opus-4.6`、`claude-sonnet-4.6`、`claude-haiku-4.5`、`gpt-4o`、`gpt-4.1`）。

但從產品語意上看，模型名稱代表兩件事：
1. App 已成功連線到 SDK Server
2. App 已成功取得可用模型清單

若尚未連線就顯示模型名稱，會讓使用者誤以為目前已連線且該模型可用。同時，靜態 fallback 清單需手動維護，且與實際 Server 可用模型可能不一致，進一步加深誤導。

---

## 決策一：模型選擇器改為條件顯示

### 問題

模型選擇器是否仍應「常駐顯示」？

### 評估

| 方案 | 優點 | 缺點 |
|------|------|------|
| 常駐顯示 + 靜態 fallback（原方案）| 使用者隨時看到模型名稱 | 誤導使用者已連線；fallback 清單需手動維護 |
| **條件顯示（採用）** | 模型名稱 = 已連線的明確信號；無需維護 fallback | 未連線時工具列空間較空（可接受，已有連線狀態提示） |

### 結論

模型選擇器的顯示條件改為：**`GET /models` 成功回傳非空清單後才顯示**。

- `disconnected` / `connecting` / `error` 狀態 → 不顯示模型選擇器
- 連線成功後 `GET /models` 回傳空清單或失敗 → 不顯示
- 連線成功且 `GET /models` 回傳非空清單 → 顯示

推理力度選擇器依附於模型選擇器，因此在模型選擇器不顯示時，推理力度選擇器也不顯示（既有邏輯，無需額外調整）。

### 理由

- 模型名稱是「連線成功」的自然信號，不應在未確認前顯示
- 既有 TopBar 連線狀態 Chip 已提供連線狀態反饋（灰點 + 「未連線」），工具列不必重複承擔此職責
- 減少維護靜態 fallback 清單的負擔

---

## 決策二：移除靜態 fallback 清單

### 問題

既有的「靜態 fallback 模型清單」是否仍需保留？

### 評估

| 方案 | 優點 | 缺點 |
|------|------|------|
| 保留 fallback 作為 UI 預設 | 離線時仍可展示模型清單 | 清單過時風險高；與決策一「不顯示」衝突 |
| **移除 fallback（採用）** | 單一資料來源（`GET /models`）；無維護負擔 | 無（決策一已決定不顯示） |

### 結論

從 UI 規格中**完全移除靜態 fallback 清單**。模型清單的唯一來源為 `GET /models` API。

### 理由

- 決策一已確立未連線時不顯示選擇器，靜態清單失去用途
- 清單由 Server 動態提供，避免 spec 與實際模型不同步

---

## 決策三：連線中斷時回到「不顯示」狀態

### 問題

若連線中斷或 `GET /models` 重新取得失敗，應顯示上次取得的模型，還是隱藏選擇器？

### 評估

| 方案 | 優點 | 缺點 |
|------|------|------|
| 保留上次模型清單（快取） | 使用者可繼續查看模型名稱 | 模型可能已不可用，發訊息必定失敗，造成混淆 |
| **隱藏選擇器（採用）** | 模型名稱永遠對應「可用」語意 | 需要再次連線後才顯示（可接受） |

### 結論

連線中斷時，清除模型清單狀態，回到「不顯示模型選擇器」狀態。重新連線成功後再次取得模型，恢復顯示。

### 理由

- 保持「模型名稱 = 可用」的語意一致性
- 連線中斷時無法送訊息，顯示模型名稱沒有實質用途

---

## 決策四：per-role 歷史選擇的回退規則

### 問題

移除 fallback 清單後，角色歷史選擇的模型不在當前 `GET /models` 清單中時如何處理？

### 結論

回退至 `GET /models` 清單的**第一個模型**（取代原先「回退至靜態 fallback 的第一個模型」）。

首次無歷史記錄時，也使用 `GET /models` 清單的第一個模型作為預設。

### 理由

- `GET /models` 回傳順序由 Server 決定，第一個即為 Server 推薦的預設模型
- 無需維護額外的預設模型 ID
